TanStack Query의 구조와 내부 동작 원리에 대해 분석한 글이다.
어떻게 구성되었으며, 핵심 객체들의 역할을 파악하면 라이브러리를 사용할 때 도움이 될 것이다.

## TanStack Query란?
결론부터 알아보자면, `서버 상태 관리 라이브러리` 라고 할 수 있다.
> [공식 문서](https://tanstack.com/query/latest/docs/framework/react/overview#:~:text=it%20makes%20fetching%2C%20caching%2C%20synchronizing%20and%20updating%20server%20state%20in%20your%20web%20applications%20a%20breeze)에 의하면 **서버 상태를 가져오고(fetching), 캐싱하고(caching), 동기화하고(synchronizing), 업데이트(updating)** 하는 작업을 쉽게 할 수 있게 도와준다.

위 내용만으로는 실제로 어떤 역할을 하는지 이해하기 어려우므로, 간단한 예시를 통해 살펴보자.

### 서버 응답 사용하기

```js
const useData = (url) => {
  const [data, setData] = useState(null)

  useEffect(() => {
    fetch(url)
      .then((res) => res.json())
      .then((result) => setData(result))
  }, [url])

  return data
}
```
커스텀 훅 `useData()` 는 실행되었을 때 요청을 보내고, 받은 응답 값을 `data` 상태에 저장한 다음 return 한다.
```js
function App() {
  const data = useData("https...")

  return <div>{data}</div>
}
```
`App`은 훅에서 제공하는 상태를 렌더링 한다.

### 로딩과 오류

응답을 받기 전까지는 사용자에게 빈 화면이 보일 수 있고, 요청이 실패했을 때에 애플리케이션이 멈출 것에 대한 처리도 필요하다.

```js
const useData = (url) => {
  const [data, setData] = useState(null)
  const [isLoading, setIsLoading] = useState(true)
  const [isError, setIsError] = useState(false)

  useEffect(() => {
    fetch(url)
      .then((res) => res.json())
      .then((result) => setData(result))
      .catch(() => setIsError(true))
      .finally(() => setIsLoading(false))
  }, [url])

  return { data, isLoading, isError }
}
```

위 코드를 보면, 요청이 시작되기 전에는 `isLoading: true` 로 두어 아직 데이터를 불러오는 중임을 나타낸다. 요청이 성공하면 데이터를 저장하고, 실패하면 `isError: true` 로 바꿔 에러 상태를 표현한다. 

### 캐싱과 재시도

다른 컴포넌트에서도 재사용할 수 있도록 훅을 확장해 보자. 데이터 변경이 드물다면 매번 네트워크 요청을 보내기보다 캐시를 활용하는 편이 더 효율적이다.

또한 네트워크 오류는 일시적으로 발생하는 경우가 많으므로, 바로 에러 화면을 보여주기보다 몇 차례 재시도하는 방식이 더 나은 사용자 경험을 줄 수 있다.

```js
const cache = new Map() // URL을 키로 하는 캐시

const useData = (url) => {
  const [data, setData] = useState(null)
  const [isLoading, setIsLoading] = useState(true)
  const [isError, setIsError] = useState(false)

  const fetchData = (retryCount) => {
    fetch(url)
      .then((res) => res.json())
      .then((result) => {
        setData(result)
        cache.set(url, result) // 캐시에 저장
      })
      .catch(() => {
        if (retryCount > 0) {
          // 1초 후에 재요청
          setTimeout(() => fetchData(retryCount - 1), 1000)
        } else {
          setIsError(true)
        }
      })
      .finally(() => {
        setIsLoading(false)
      })
  }

  useEffect(() => {
    // 캐시가 존재한다면
    if (cache.has(url)) {
      setData(cache.get(url))
      setIsLoading(false)
      return
    }

    // 존재하지 않는다면 네트워크 요청을 한다
    fetchData(3)
  }, [url])

  return { data, isLoading, isError }
}
```

`fetchData()` 함수는 재시도 횟수를 인자로 받아 요청한다. 네트워크 요청에 실패하면 1초 후 재요청을 하고, 재시도 횟수 내에 응답을 받지 못하면 `Error`를 반환하여 일시적인 실패가 아님을 밝힌다.

만약 `Cache`가 존재한다면 네트워크 요청을 보내지 않고 존재하는 캐시 데이터를 반환하게 된다.

### 이 모든 것을 대체할 수 있는 useQuery

위 예제는 네트워크 요청을 위한 코드였다. 단순 요청만 하지 않고, 사용자 경험을 향상시키기 위한 로딩, 에러, 캐싱 데이터를 관리하고 있다. 이 모든 상태 관리를 최적화하여 처리해주는 라이브러리가 `Tanstack query` 이며, 구현한 기능은 아래 `useQuery` 로 대체할 수 있다.

```js
const useData = (url) => {
  return useQuery({
    queryKey: [url],
    queryFn: fetchFn,
    retry: 3,
    ...
  })
}
```

간단히 의미에 대해 알아보자면:
* queryKey: 캐시 키로 사용하는 고유 식별자
* queryFn: 네트워크 호출 함수
* retry: 네트워크 요청 재시도 횟수

### Redux와 다른 역할

`TanStack Query` 와 `Redux` 는 둘 다 전역으로 상태를 관리하는 역할이다.
그렇다면 둘이 무슨 차이가 있고, **"서버 상태 관리"** 라이브러리라고 불리는가?

**Redux**는 쉽게 말해 클라이언트 전역 상태 관리 라이브러리이며, 사용자 인터랙션이 트리거다. 
_(e.g. 장바구니에 상품 추가)_

**TanStack Query**는 앞서 살펴본 것처럼 네트워크 요청을 통해 갱신되며, 그에 대한 상태(네트워크 상태, 캐싱, 재시도 등)을 업데이트 하고 관리하는 방식이므로 **"서버 상태 관리"** 라고 구분 짓는 것이다.

## 아키텍처
TanStack Query는 핵심 로직을 `query-core` 패키지로 구성했고,
React에서 쉽게 사용할 수 있도록 `react-query` 패키지로 분리했다.

| 패키지       | 역할              | 주요 구성 요소                                              |
|--------------|------------------|-------------------------------------------------------------|
| query-core   | 핵심 로직 | `QueryObserver`, `Query`, `QueryCache`, `QueryClient`       |
| react-query  | 프레임워크 별 지원 | `useQuery`, `useBaseQuery` 등                      |

### 단순한 useQuery의 구조

`react-query` 패키지에서 제공하는 훅이며, 실제로는 내부 구조가 매우 단순하다. ([실제 코드 링크](https://github.com/TanStack/query/blob/main/packages/react-query/src/useQuery.ts))

```js
export function useQuery(options, queryClient) {
  return useBaseQuery(options, QueryObserver, queryClient)
}
```

`useBaseQuery()` 와 `QueryObserver` 
그리고 `Query, QueryCache, QueryClient`로 이어지는 핵심 객체 위주로 살펴보자.


## useBaseQuery() 역할

`useBaseQuery()`는 `QueryClient` 를 받아서 `QueryObserver`를 만들고, 이를 React와 연결하고 최종적으로 구독 가능한 결과를 반환하는 핵심 연결 지점이다.

코드를 분석해보기 전에 전체 데이터 흐름 과정을 보자.

1. `useQuery` 를 통해 `useBaseQuery`가 실행되면 먼저 `QueryObserver`를 생성한다.
2. `QueryObserver`는 `QueryCache`에서 현재 `Query`를 찾거나 새로 만든다.
3. 이미 `Query`가 있으면 기존 상태를 재사용하고, 없으면 새 `Query` 인스턴스를 생성한다.
4. 이때 캐시 데이터가 살아 있고 stale 상태가 아니라면, 기존 캐시 데이터를 그대로 사용한다.
5. 반대로 데이터가 없거나 stale 상태라면 `Query.fetch()`가 실행되어 네트워크 요청을 보낸다.
6. 요청이 성공하면 `Query`의 상태가 업데이트되고, 실패하면 에러 상태가 저장된다.
7. 상태가 바뀌면 `QueryObserver`가 변경을 전달받는다.
8. 이후 `notifyManager`가 알림을 batch 처리하고, `useSyncExternalStore`를 통해 React 컴포넌트와 동기화한다.
9. 최종적으로 컴포넌트는 최신 `data`, `error`, `status`를 기준으로 다시 렌더링된다.


### QueryClient 획득
> 💡 TanStack Query의 전체 쿼리 상태를 관리하는 중앙 관리자다.  QueryCache, 기본 옵션, 동작 등을 제어한다.

`useBaseQuery()` 한 줄 설명대로 내부 코드를 확인해보면,

```js
// https://github.com/TanStack/query/blob/main/packages/preact-query/src/useBaseQuery.ts#L54
const client = useQueryClient(queryClient)
const defaultedOptions = client.defaultQueryOptions(options)
```

넘겨받은 queryClient나 기본 Context로 제공되는 값을 사용한다.
options 인자에 기본 옵션을 더해둔다.

### QueryObserver 생성
> 💡 특정 Query를 구독하면서 그 결과를 컴포넌트에 맞는 형태로 전달하는 객체다.
Query 상태가 바뀌면 이를 감지해 React 컴포넌트가 다시 렌더링되도록 연결해준다

```js
// https://github.com/TanStack/query/blob/main/packages/preact-query/src/useBaseQuery.ts#L84
const [observer] = useState(
  () =>
  new Observer<TQueryFnData, TError, TData, TQueryData, TQueryKey>(
    client,
    defaultedOptions,
  ),
 )
```
`client`와 `options`로 Observer를 만든다. 쿼리 상태를 추적하고 변경을 알리는 역할이다.
즉, Query와 Component 사이의 다리 역할이다.

## 비교 과정 (React 내부에서)

1. **렌더링 시점**에 `fiber.memoizedState`를 순회하며 훅 정보를 순서대로 저장한다.
2. **deps 비교**는 다음의 순서로 수행된다:
    1. 이전 deps가 없는 경우 (첫 실행 또는 `null`) → 무조건 실행
    2. 이전 deps와 현재 deps의 **길이가 다르면** → 무조건 실행
    3. 둘 다 아니면, 각 요소를 반복문으로 돌면서 `Object.is(prev[i], next[i])`로 비교
        
        → 하나라도 다르면 실행

# 📌 피연산자에 따른 명령어 형식

### 🧾 스택 구조: 0-주소 명령어 형식 (Stack Machine)

- 0-주소 명령어 형식
    
    ## ✅ 스택 머신이란?
    
    > Stack Machine은 연산을 위해 스택(Stack) 자료구조를 사용하는 컴퓨터 구조를 말한다.
    > 
    
    이 구조에서 사용되는 명령어는 **0-주소 명령어 형식(zero-address instruction format)**이다.
    
    명령어에 `opcode`만 존재하고, `피연산자`가 없는 형식이다.
    
    **예시:**
    
    ```nasm
    ADD
    ```
    
    보통 A라는 값을 B에 더해야 하는데, 위 코드에서 2개의 피연산자는 어디에 숨어있는가?
    그건 스택에 있다.
    
    → 즉, 연산 대상은 모두 **스택에 저장된 값들**이다.
    
    ---
    
    ## ✅ 예제: 수식 변환
    
    ```nasm
    D = A + B × C
    ```
    
    위 수식을 **스택 머신 방식의 어셈블리어(0-주소 명령어)**로 바꾸면 다음과 같다:
    
    ```nasm
    
    PUSH A
    PUSH B
    PUSH C
    MUL
    ADD
    POP D
    ```
    
    ---
    
    ## ✅ 명령어 해석
    
    ### 1. `PUSH A`, `PUSH B`, `PUSH C`
    
    - 스택에 차례로 **A, B, C 값을 쌓는다.**
    - 스택 상태 (아래가 바닥):
        
        ```
        
        ┌───────┐ ← Top
        │   C   │
        │   B   │
        │   A   │
        └───────┘
        ```
        
    
    ### 2. `MUL`
    
    - 스택 상단의 두 값 `C`, `B`를 꺼낸다 (POP)
    - 두 값을 곱한 결과 `B × C`를 다시 스택에 PUSH한다.
    - 스택 상태:
        
        ```
        ┌──────────┐ ← Top
        │  B × C   │
        │    A     │
        └──────────┘
        ```
        
    
    ### 3. `ADD`
    
    - 다시 상단의 두 값 `B × C`와 `A`를 꺼낸다 (POP)
    - 더한 결과 `A + (B × C)`를 PUSH한다.
    - 스택 상태:
        
        ```
        
        ┌──────────────┐ ← Top
        │ A + (B × C)  │
        └──────────────┘
        ```
        
    
    ### 4. `POP D`
    
    - 스택 상단 값을 꺼내서 `D`라는 변수에 저장한다.
        
        → 최종 결과가 변수 `D`에 들어가게 된다.
        
    
    ---
    
    ## ✅ 요약
    
    - 스택 머신에서는 명령어에 **피연산자가 포함되지 않는다.**
    - 연산에 필요한 값은 **모두 스택에 존재**한다.
    - 연산 순서는 스택 구조(LIFO)에 따라 결정된다.
    - `PUSH`는 값을 스택에 넣고, `POP`은 스택에서 꺼낸다.
    - 연산 명령어(예: `MUL`, `ADD`)는 **상단 2개의 피연산자를 POP하고 연산 후 결과를 다시 PUSH**한다.
